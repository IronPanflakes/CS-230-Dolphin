# CS-230-Dolphin
CS 230 Design Documentation
This project involved designing a complete software architecture for The Gaming Room, a client seeking to expand their existing game, Draw It or Lose It, into a scalable, web‑based, multi‑platform application. 
Their requirements centered on supporting multiple concurrent game sessions, enforcing unique team and player identities, and ensuring the system could run reliably across different operating platforms. 
The goal was to produce a design document that clearly communicated the application’s structure, constraints, and recommended technologies to both the client and a development team.
One area I executed particularly well was the clarity and organization of the design documentation. I translated the client’s requirements into concrete design constraints and architectural decisions, and I articulated the reasoning behind each choice in a way that would be understandable to both technical and non‑technical stakeholders. 
The evaluation of operating platforms and the final recommendations section were especially strong, providing a balanced, well‑supported comparison that aligned with the client’s needs.
Working through the design document before writing any code was surprisingly helpful. It forced me to think through the system’s structure, data flow, and constraints early, which made the later coding tasks more straightforward. 
Having a clear domain model and understanding the relationships between objects reduced ambiguity and helped ensure the implementation aligned with the intended architecture.
If I were to revise one part of the document, I would refine the domain model section to include more explicit examples of how the classes interact during gameplay. 
While the UML and descriptions were accurate, adding a short sequence diagram or more detailed interaction notes would make the design even easier for a future developer to pick up and implement.
Interpreting the user’s needs required translating their high‑level goals—such as “support multiple teams” or “ensure unique game instances”—into specific technical requirements like enforcing singleton patterns, managing shared resources, and preventing naming collisions. 
Considering user needs is essential in software design because it ensures the final product is intuitive, functional, and aligned with real‑world use cases rather than assumptions.
My approach to designing the software was grounded in modularity, scalability, and clarity. I relied on techniques such as identifying constraints early, using UML to visualize the system, and evaluating platforms based on objective criteria like performance, cost, and compatibility. 
In future projects, I would continue using structured design documents, platform evaluations, and early modeling, as these strategies create a strong foundation for both development and communication with stakeholders.
